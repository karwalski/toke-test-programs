# Toke Worker Infrastructure — Automated Program Generation Pipeline

This directory contains everything needed to provision, run, and monitor a fleet
of 5 Lightsail workers that generate toke programs from requirements.

## Architecture

```
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  Your Machine   │────▶│  5x Lightsail    │────▶│ api.tokelang.dev│
│  (orchestrator) │     │  Workers         │     │ (toke generation)│
└─────────────────┘     └──────────────────┘     └─────────────────┘
                              │                        │
                              │                  ┌─────────────────┐
                              └─────────────────▶│ Anthropic API   │
                                                 │ (repair loop)   │
                                                 └─────────────────┘
```

Each worker:
1. Reads requirements from `categories/*/requirements.yaml`
2. Partitions work by `hash(id) % 5` so no overlap between workers
3. Calls the toke API for initial generation
4. Compiles with `tkc --check`
5. Runs test cases
6. On failure: enters repair loop via Claude (up to 5 iterations)
7. Saves successful solutions + companion docs

## Files

| File | Purpose |
|------|---------|
| `worker-setup.sh` | Lightsail user-data bootstrap script |
| `worker-generate.py` | Main generation loop (runs on each worker) |
| `worker-companion.py` | Generates .tkc.md documentation files |
| `orchestrator.py` | Local monitoring/control tool |
| `collect-results.sh` | Pull solutions from workers to local repo |
| `launch-workers.sh` | Create the 5 Lightsail instances (AWS CLI) |
| `workers.json` | Worker ID → IP mapping (generated by launch) |

## Workflow

### 1. Provision (when ready)

```bash
# Creates 5 Lightsail Small instances in ap-southeast-2
bash infra/launch-workers.sh
```

### 2. Configure

SSH into each worker and set the real API keys:

```bash
ssh -i ~/.ssh/toke-workers.pem ubuntu@<IP>
sudo vi /opt/toke-worker/.env
# Set: ANTHROPIC_API_KEY, TOKE_API_KEY, WORKER_ID (1-5)
```

### 3. Run

```bash
# Start all workers
python3 infra/orchestrator.py start

# Or start a specific worker
python3 infra/orchestrator.py start 3
```

### 4. Monitor

```bash
# One-shot status
python3 infra/orchestrator.py status

# Continuous watch (refreshes every 60s)
python3 infra/orchestrator.py watch
```

### 5. Collect Results

```bash
# Pull all completed solutions to local repo
bash infra/collect-results.sh

# Review and commit
git status
git add categories/
git commit -m "Add generated solutions from workers"
```

### 6. Stop / Cleanup

```bash
# Stop workers
python3 infra/orchestrator.py stop

# Delete instances (manual via AWS console or CLI)
```

## Rate Limiting

- Each worker sleeps 1 second between API calls
- 5 workers = ~5 req/sec total to api.tokelang.dev
- Anthropic API calls only happen during repair (failures)

## Resumability

Workers are fully resumable. `state.json` tracks every completed/failed/skipped
requirement. If a worker crashes or is restarted, it picks up where it left off.

## Security

- `.env` files contain API keys and are chmod 600
- `.env` is in `.gitignore` — never committed
- `workers.json` contains IPs — also gitignored
- SSH keys stored at `~/.ssh/toke-workers.pem`
- Only port 22 open on worker instances

## Cost Estimate

- 5x Lightsail Small: ~$50/month total ($10/instance)
- Anthropic API: varies by repair rate (~$0.003/repair call)
- Toke API: included in tokelang.dev plan
